# Bullvaan Codebase Architecture Analysis

## Executive Summary

**Overall Assessment:** ✅ **User data is PROPERLY ISOLATED** (database-level) **BUT** ⚠️ **has architectural constraints with auto-trader singleton pattern that needs documentation**

The system uses a solid multi-user architecture with database-enforced per-user data isolation, but there are important limitations regarding concurrent auto-trader usage that must be understood.

---

## 1. User Account Creation & Storage

### Flow: Frontend → Backend → Supabase

```
Frontend (Signup.js) → /api/signup (auth_routes.py) → Supabase users table
```

### Implementation Details:

- **Signup Endpoint:** `POST /api/signup` (in [api/auth_routes.py](api/auth_routes.py#L44))
  - Accepts: `email`, `password`
  - Returns: JWT token, user_id, email
- **Database Schema** ([database/models.py](database/models.py#L12)):

  ```python
  User table:
  - id (String, primary key, auto-generated by Supabase)
  - email (String, unique)
  - password (String, bcrypt hashed)
  - created_at (DateTime)
  - updated_at (DateTime)
  ```

- **Password Security:** ✅ Bcrypt hashing with 12 rounds (see [utils/auth.py](utils/auth.py#L23))

- **Frontend Storage:**
  - ✅ `access_token` (JWT, 24-hour expiration)
  - ✅ `user_id` (extracted from JWT payload)
  - ✅ `email` (extracted from JWT payload)
  - ⚠️ Stored in localStorage (NOT httpOnly cookies - acceptable for SPA but noted as "would be more secure in production")

## 2. Trade Storage & User Association

### Architecture: Per-User Isolation ✅

**Storage Location:** Supabase `trades` table (not JSON file)

**Database Schema** ([database/models.py](database/models.py#L45)):

```python
Trade table:
- id (String, primary key)
- user_id (String, FOREIGN KEY → users.id) ← KEY TO ISOLATION
- date (String, YYYY-MM-DD)
- name (String, e.g., "NIFTY 25400 CE")
- quantity, lot, buy_price, sell_price
- pnl, status (open/closed)
- buy_time, sell_time
- auto (Boolean - auto-trader generated)
- mode (paper/real)
- created_at, updated_at
```

### API Endpoints (All User-Gated):

1. **GET /trades** ([server.py](backend/api/server.py#L1833))
   - Uses `Depends(get_current_user)` ✅
   - Filters: `get_user_trades_by_date(user_id, date)` ([trades_db.py](backend/utils/trades_db.py#L54))
   - Returns only current user's trades

2. **POST /trades** ([server.py](backend/api/server.py#L1853))
   - Extracts user_id from JWT
   - Sets `trade['user_id'] = user_id` before saving ✅
   - Cannot create trades for other users

3. **PUT /trades/{trade_id}** ([server.py](backend/api/server.py#L1950))
   - Verifies ownership: `existing_trade.get('user_id') != user_id` → 401 error ✅
   - Only trade owner can update

4. **DELETE /trades/{trade_id}** ([server.py](backend/api/server.py#L2006))
   - Calls `delete_trade(trade_id, user_id)`
   - Verifies: `trade.get('user_id') != int(user_id)` → rejects ✅

### Data Isolation: ✅ COMPLETE

- Every trade query filtered by `user_id` at database level
- Frontend cannot bypass via API (JWT required on all endpoints)
- Unauthorized user access returns 401/403

## 3. Auto-Trader User Identification

### Critical Design: Global Auto-Trader, Per-User Context ⚠️

**Challenge:** The auto-trader engine runs as a single background task in the FastAPI server. Only ONE auto-trader instance can run per server instance.

### User Context Management:

**Files:**

- [backend/utils/auto_trader_db.py](backend/utils/auto_trader_db.py#L8)
- [backend/engine/auto_trader.py](backend/engine/auto_trader.py#L54)

**Flow:**

```
1. User calls POST /auto-trader/start with JWT
2. Server.py extracts user_id from JWT ✅
3. Sets global context:
   - auto_trader_module._current_user_id = user_id
   - set_auto_trader_user_id(user_id)  # for trades_db functions
   - _autotrader_user_id = user_id      # for server.py tracking
4. Auto-trader generates trades via save_auto_trade()
5. save_auto_trade() injects user_id before saving ✅
```

### Start Endpoint ([server.py](backend/api/server.py#L2047)):

```python
@app.post("/auto-trader/start")
async def auto_trader_start(current_user: dict = Depends(get_current_user)):
    user_id = current_user.get('user_id')

    # If already running for different user, STOP IT FIRST
    if auto_trader.enabled and _autotrader_user_id != user_id:
        logger.warning(f"Stopping auto-trader for user {_autotrader_user_id}")
        auto_trader.stop()

    # Set context and start
    set_auto_trader_user_id(user_id)
    auto_trader.start(loop=loop)
    return {"status": "started", "user_id": user_id}
```

### Stop Endpoint ([server.py](backend/api/server.py#L2078)):

```python
# Only user who started it can stop it ✅
if _autotrader_user_id and _autotrader_user_id != user_id:
    return {"status": "error", "detail": "Running for another user"}
```

### User Identification: ✅ CORRECT (but architecturally limited)

- ✅ Each auto-trade tagged with correct user_id
- ✅ JWT determines which user is trading
- ⚠️ **Limitation:** Cannot run multiple users' auto-traders simultaneously

## 4. Authentication & Session Management

### JWT Token Flow:

**Components:**

- Token Creation: [utils/auth.py](backend/utils/auth.py#L41)
- Token Verification: [utils/auth.py](backend/utils/auth.py#L51)
- FastAPI Dependency: [utils/auth.py](backend/utils/auth.py#L93)

**Token Structure:**

```python
{
  "user_id": "12345",
  "email": "user@example.com",
  "iat": 1708123456,  # issued at
  "exp": 1708209856   # expires in 24 hours
}
```

**Verification:** Every API endpoint uses `Depends(get_current_user)`

- Extracts `Authorization: Bearer <token>` header
- Validates signature (JWT_SECRET)
- Checks expiration
- Returns user_id and email

### Security Measures:

✅ Bcrypt password hashing (12 rounds)  
✅ JWT token with 24-hour expiration  
✅ User extraction from JWT on every request  
✅ Authorization header required on all protected endpoints  
✅ User ownership verification on trade operations

### Vulnerabilities (Minor):

⚠️ JWT stored in localStorage instead of httpOnly cookie (noted in code)  
⚠️ No token refresh endpoint implementation (though auth_routes.py has skeleton)

## 5. Trade Data: Global vs Per-User

### Current State: ✅ PER-USER (Properly Isolated)

**Supabase Database:**

- Each trade record has explicit `user_id` foreign key
- Queries always filter by `user_id`
- Database schema enforces foreign key constraint

**Legacy Code:** ⚠️ trades.json exists but appears to be fallback/caching only

- [backend/data/trades.json](backend/data/trades.json) - contains test data
- [backend/engine/auto_trader.py](backend/engine/auto_trader.py#L58) - has `_load_trades()` / `_save_trades()` functions
- These seem to be caching mechanisms, actual persistence is Supabase

### Data Flow:

```
Frontend → GET /trades → get_user_trades_by_date(user_id, date)
                       ↓
                   Supabase query with WHERE user_id = ?
                       ↓
                   Returns only user's trades
```

Each user sees only their own trades. ✅ CONFIRMED

## 7. Potential Issues & Considerations

### ✅ CORRECTLY IMPLEMENTED:

1. **User Data Isolation** - Database-enforced per-user trade records
2. **Authentication** - JWT tokens with proper validation
3. **Authorization** - Every endpoint checks user ownership
4. **Password Security** - Bcrypt hashing
5. **User Context in Auto-Trader** - Correct user_id tagging

### ⚠️ ARCHITECTURAL CONSTRAINTS (Not bugs, but design limitations):

1. **Auto-Trader Singleton Pattern**
   - Only ONE auto-trader runs at a time per server instance
   - When user B starts auto-trader while user A is running → stops user A's trader
   - **Mitigation:** Document this clearly. Users must coordinate. Or deploy separate worker pods per user.
2. **trades.json Legacy Code**
   - System has both `trades.json` file AND Supabase database
   - `auto_trader.py` has `_load_trades()` / `_save_trades()` for JSON
   - Actual persistence is Supabase (database)
   - `trades_db.py` queries Supabase directly
   - **Mitigation:** Remove JSON file operations, ensure all paths use Supabase

### ⚠️ SECURITY CONSIDERATIONS:

1. **JWT in localStorage**
   - Currently stored in browser localStorage (not httpOnly)
   - Vulnerable to XSS attacks
   - **Mitigation:** Use httpOnly cookies in production
2. **Login endpoint discrepancy**
   - Old [api/login.py](api/login.py) doesn't return JWT
   - Proper [api/auth_routes.py](api/auth_routes.py) does return JWT
   - **Status:** Actually okay - server.py uses auth_routes.py
   - **Mitigation:** Delete old login.py and signup.py files to avoid confusion

3. **No Rate Limiting**
   - No rate limiting on /api/login or /api/signup
   - **Mitigation:** Add rate limiting middleware

## Conclusion

**The codebase demonstrates proper user data isolation at the database level.** All trades are correctly associated with users through foreign key relationships, and API endpoints enforce user ownership checks.

**The primary architectural constraint is the auto-trader singleton pattern**, which prevents simultaneous auto-trading for multiple users. This is a design choice, not a bug, but should be clearly documented.

**No critical shared state issues were found.** Data separation is complete and enforced at multiple layers (database schema, API authentication, authorization checks).
